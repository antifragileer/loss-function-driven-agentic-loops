# Security Policy

## Reporting a vulnerability

**Please do not open a public GitHub issue for security
vulnerabilities.**

Send a private report to
[security@antifragileer.dev](mailto:security@antifragileer.dev)
with:

- A description of the vulnerability
- Steps to reproduce
- The affected component (skill, parser, wrapper, or
  installer) and version
- The bundle version (from `./install.sh --list` or
  `bundle.json`)

We will acknowledge receipt within 72 hours and aim to
ship a fix within 14 days for critical issues, 30 days
for high/medium issues. We follow coordinated disclosure:
we'll work with you on a disclosure timeline.

## Supported versions

Only the **latest minor** of the bundle receives security
fixes. Older minor versions are not patched; users should
upgrade via `./install.sh --force`.

| Bundle version | Supported?          |
|----------------|---------------------|
| 2.x (latest)   | ✅ yes              |
| 1.x            | ❌ no — upgrade to 2.x |

## Threat model

This bundle is a **distribution of agent skills**. It is
**not** a network service, **not** a daemon, and **does
not** execute code on the user's machine outside the
explicit `./install.sh` flow. The threat surface is:

1. **The installer** (`install.sh`) — runs as the user,
   `cp -R`s skills into a profile's `skills/` directory.
   We audit this script on every change.
2. **The wrapper scripts** (the `*-wrapper.sh` files
   generated by `harness-scaffold` per project, NOT
   shipped in this repo) — invoke a coding agent in a
   per-cycle directory. Out of scope for this repo.
3. **The parsers** (`scripts/parse_*.py`) — read the
   agent's output file and emit JSON. They run as the
   user; we audit the input handling.
4. **The bundled skills** — markdown documentation +
   Python parsers. The risk is that a malicious change
   could exfiltrate data via a parser reading an
   unexpected file. We review parsers carefully.

Out of scope:

- The coding agents themselves (Cline, Claude Code,
  Codex, Hermes, OpenCode). Report issues to those
  projects.
- The `hermes-agent` install — Hermes is its own
  project. Report issues at
  https://github.com/NousResearch/hermes-agent.

## Security best practices for users

1. **Pin your bundle version.** Use a specific tag (e.g.
   `git checkout v2.0.0`), not `main`, in production
   loops.
2. **Audit the wrapper scripts** the scaffold writes
   into your project. The scaffold generates
   `verifiers/<runtime>-wrapper.sh` from the
   adapter skill's contract; review it before running
   cycles.
3. **Use the `drift` sub-loss** to catch model drift —
   if the inner agent's `model` field changes, the loop
   catches it.
4. **Use a hermetic per-iteration directory.** The
   scaffold seeds `.iterations/<cycle>/` with the
   project source and excludes `.git/` and
   `.iterations/`. Don't override this in production.
5. **Review cycle artifacts.** `logs/cycle-N/` contains
   the wrapper output, the parser output, the
   sub-losses, and the design-set score. Inspect them.
6. **Don't run cycles as root.** The wrapper scripts
   need write access to the project tree; running as
   an unprivileged user is sufficient.

## Cryptographic integrity (future)

We're tracking a `checksums.txt` and signed release
process for v3.0.0. Until then, the bundle is shipped
over HTTPS and the user verifies the repo URL matches
`https://github.com/antifragileer/loss-function-development-skills`.
