# Templates

The `templates/` directory in the harness-scaffold skill holds
the templates the scaffold uses to generate files. They are
inlined as Python string constants in `scripts/scaffold.py`
because they need to be embedded in the script's execution
context (the scaffold runs as a single self-contained Python
file, no separate template-loading needed).

If you need to override a template (e.g., to change the AGENTS.md
operating rules or the design task stub format), the templates
are in `scripts/scaffold.py` as module-level string constants:

To find a specific template, grep for the constant name in
`scripts/scaffold.py`.

## Why templates are inlined, not separate files

Three reasons:

1. **Distribution simplicity.** The skill is one file
   (`SKILL.md`) plus one script (`scripts/scaffold.py`). Users
   drop these into their `skills/harness-scaffold/` directory
   and they work. If templates were separate, the skill would
   have 18+ more files.
2. **No template-loading bugs.** No `Path(__file__).parent / "templates" / "..."` to break, no Unicode escape mismatches between editor and runtime, no encoding issues.
3. **The user can still customize.** The constants are
   module-level and overrideable. A power user can `import
   scaffold; scaffold.AGENTS_MD = my_custom_agents_md` and
   re-run the main function with their own templates.

The downside is that the script is large (~815 lines). The
upside is that it's portable, debuggable, and doesn't depend
on the file system at runtime.

## If you need a separate file

The pattern for moving a template to its own file:

```python
# Before
AGENTS_MD = lambda project_name: (
    "# ...\n"
    + "# " + project_name + "\n"
    ...
)

# After
def AGENTS_MD(project_name: str) -> str:
    tmpl = (Path(__file__).parent / "templates" / "agents.md.j2").read_text()
    return tmpl.replace("{{ project_name }}", project_name)
```

We deliberately don't ship this. The inlined approach is the
one we tested and verified across 5 cycles of smoke-testing
the scaffold. Switching to file-based templates is a v2 task.

## Testing a template change

After editing any template constant:

```bash
# Re-run the scaffold on a fresh directory
rm -rf /tmp/_scaffold_smoke
harness-scaffold \
  --project-root /tmp/_scaffold_smoke \
  --goal-prompt /path/to/GOAL.md

# Verify the file you changed
cat /tmp/_scaffold_smoke/AGENTS.md  # or whatever you changed
```

Re-running the scaffold on an existing project is safe — files
without the `agent-edited: yes` marker are overwritten. Files
with the marker are preserved.
